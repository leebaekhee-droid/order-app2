# 커피 주문 앱

## 1. 프로젝트 개요

### 1.1 프로젝트 명 
커피 주문 앱

### 1.2 프로젝트 목적
사용자가 커피 메뉴를 주문하고, 관리자가 주문을 관리 할 수 있는 간단한 풀스택 웹 앱

### 1.3 개발 범위
- 주문하기 화면(메뉴 선택 및 장바구니 기능)
- 관리자 화면 (재고 관리 및 주문 상태 관리)
- 데이터를 생성/조회/수정/삭제할 수 있는 기능

## 2. 기술 스택
- 프런트 엔드: HTML, CSS, 리액트, 자바스크립트
- 백엔드: Node.js, Express
- 데이터베이스: PostgreSQL

## 3. 기본 사항
- 프런트엔드와 백엔드를 따로 개발
- 기본적인 웹 기술만 사용
- 학습 목적이므로 사용자 인증이나 결제 기능은 제외
- 메뉴는 커피 메뉴만 있음

## 4. 프런트엔드 화면 PRD

### 4.1 공통
- 상단 공통 앱 바에 서비스 로고 텍스트 `COZY`와 `주문하기`, `관리자` 버튼을 배치한다.
- 화면 진입 시 기본 화면은 `주문하기` 화면으로 한다.
- 버튼 스타일, 폰트, 여백 규칙은 공통 디자인 시스템을 따른다.

### 4.2 주문하기 화면

#### 4.2.1 목적 / 목표
- 사용자가 메뉴를 선택하고 장바구니에 담은 뒤, 쉽게 주문을 완료할 수 있는 화면 제공.
- 메뉴 선택과 옵션 추가를 3~5번 이내의 조작으로 완료할 수 있도록 단순한 UX 제공.
- 현재 장바구니와 총 금액을 항상 한눈에 파악 가능하게 표시.

#### 4.2.2 대상 사용자 / 시나리오
- 카페 매장을 방문한 일반 고객, 또는 스태프가 대신 주문을 입력하는 경우.
- 대표 시나리오
  1. 앱을 열면 기본으로 `주문하기` 화면이 열린다.
  2. 상단에서 `COZY` 로고와 `주문하기 / 관리자` 버튼을 확인한다.
  3. 사용자가 원하는 음료 카드를 선택해 옵션(샷 추가, 시럽 추가 등)을 체크한다.
  4. `담기` 버튼을 눌러 장바구니에 추가한다.
  5. 여러 메뉴를 반복해서 담는다.
  6. 하단 `장바구니` 영역에서 담긴 품목, 수량, 금액과 **총 금액**을 확인한다.
  7. 필요 시 장바구니에서 수량을 조정하거나 품목을 제거한다.
  8. `주문하기` 버튼을 눌러 주문을 확정한다.

#### 4.2.3 정보 구조 / 레이아웃
- 상단 앱 바
  - 좌측: `COZY` 텍스트 로고.
  - 우측: `주문하기`, `관리자` 버튼. 현재 화면인 `주문하기`는 시각적으로 활성 상태 표시.
- 메뉴 리스트 영역
  - 여러 개의 메뉴 카드가 가로 또는 세로 그리드로 정렬된다(반응형).
  - 각 메뉴 카드 구성
    - 이미지 영역(썸네일).
    - 메뉴 이름 예: `아메리카노 (ICE)`, `아메리카노 (HOT)`, `카페라떼`.
    - 가격 예: `4,000원`, `5,000원`.
    - 간단한 설명 텍스트(1~2줄).
    - 옵션 체크박스
      - `샷 추가 (+500원)`
      - `시럽 추가 (+50원)`
    - `담기` 버튼: 클릭 시 현재 선택된 옵션 상태로 장바구니에 추가.
- 장바구니 영역
  - 타이틀: `장바구니`.
  - 품목 리스트: 각 품목은 `상품명 / 옵션 요약 / 수량 / 금액` 정보로 구성.
  - 하단에 `총 금액`과 합계 금액을 굵게 표시.
  - 우측 혹은 하단에 `주문하기` 버튼을 배치.

#### 4.2.4 기능 요구사항
- 메뉴 카드
  - 메뉴 데이터 필드: `id`, `name`, `temperature`, `basePrice`, `description`, `imageUrl(optional)`, `options[]`.
  - 옵션 데이터 필드: `id`, `label`, `extraPrice`.
  - 옵션은 체크박스로 다중 선택 가능하며, `담기` 버튼 클릭 시 해당 옵션이 반영된다.
- 장바구니
  - 품목 추가
    - `담기` 클릭 시 선택된 메뉴 + 옵션 조합이 장바구니에 추가된다.
    - 동일한 메뉴 + 동일 옵션 조합인 경우 수량을 증가시키는 정책을 기본으로 한다.
  - 수량 변경
    - 각 품목에 `+ / -` 버튼 또는 수량 조절 UI를 제공한다.
    - 최소 수량은 1이며, 0이 되면 품목을 삭제한다.
  - 삭제
    - 별도 `삭제` 버튼 또는 수량 0 처리로 항목 삭제를 지원한다.
  - 금액 계산
    - 품목 금액 = (기본 가격 + 선택 옵션 추가 금액 합) × 수량.
    - 총 금액 = 모든 품목 금액의 합.
    - 수량/옵션 변경 시 즉시 합계 갱신.
- 주문하기 버튼
  - 장바구니가 비어 있으면 비활성화 상태로 표시한다.
  - 장바구니에 1개 이상 품목이 있을 때 활성화된다.
  - 클릭 시 현재 장바구니 스냅샷을 기준으로 주문 요청을 보낸다(백엔드 연동은 별도 API 명세 참고).

#### 4.2.5 UX / UI 요구사항
- 가격과 총 금액은 굵게, 숫자는 우측 정렬한다.
- 장바구니 영역은 카드 또는 배경 색상으로 메뉴 리스트와 시각적으로 구분한다.
- `담기` 클릭 시 장바구니 영역에 짧은 하이라이트 또는 토스트 메시지를 표출한다.
- PC에서는 와이어프레임처럼 메뉴 카드 3개 가로 배치, 모바일에서는 1~2열로 재배치한다.

#### 4.2.6 상태 및 에러 처리
- 빈 장바구니
  - `장바구니가 비어 있습니다. 메뉴를 선택해 주세요.` 문구와 총 금액 0원을 표시한다.
  - `주문하기` 버튼은 비활성화 상태.
- 데이터 로딩 실패
  - 메뉴 목록 불러오기 실패 시 에러 메시지와 `다시 시도` 버튼을 제공한다.
- 네트워크/주문 실패
  - 주문 요청 실패 시 사용자에게 오류 메시지와 재시도 옵션을 제공한다.

#### 4.2.7 비기능 요구사항
- 메뉴 리스트 렌더링과 장바구니 업데이트는 200ms 이내 반응을 목표로 한다.
- 모든 버튼과 체크박스는 키보드 포커스 및 스크린 리더 사용을 고려한다.

### 4.3 관리자 화면

#### 4.3.1 목적 / 목표
- 카페 운영자가 재고를 관리하고, 들어온 주문의 상태를 실시간으로 모니터링·처리할 수 있는 화면 제공.
- 현재 재고와 주문 진행 상황을 한 화면에서 요약해 보여주는 관리자 대시보드 역할 수행.

#### 4.3.2 대상 사용자 / 시나리오
- 매장 관리자, 바리스타 등 매장 운영 스태프.
- 대표 시나리오
  1. 상단 공통 앱 바에서 `관리자` 버튼을 눌러 관리자 화면으로 이동한다.
  2. `관리자 대시보드`에서 총 주문 수, 주문 접수 수, 제조 중 수, 제조 완료 수를 확인한다.
  3. `재고 현황` 섹션에서 각 메뉴의 현재 재고를 확인하고 필요 시 `- / +` 버튼으로 조정한다.
  4. `주문 현황` 목록에서 최신 주문을 확인하고, 제조 완료 시 `주문 완료`/`주문 잔료` 버튼으로 상태를 변경한다.

#### 4.3.3 정보 구조 / 레이아웃
- 상단 앱 바
  - 좌측: `COZY` 로고 텍스트.
  - 우측: `주문하기`, `관리자` 버튼. `관리자`는 활성 상태로 표시.
- 관리자 대시보드 영역
  - 타이틀: `관리자 대시보드`.
  - 요약 텍스트 예시: `총 주문 1 / 주문 접수 1 / 제조 중 0 / 제조 완료 0`.
  - 각 수치는 향후 개별 배지/카드로 확장 가능.
- 재고 현황 영역
  - 타이틀: `재고 현황`.
  - 메뉴별 재고 카드가 가로 또는 그리드 형태로 배치된다.
  - 각 카드 구성
    - 메뉴 이름 예: `아메리카노 (ICE)`.
    - 현재 재고 수량 예: `10개`.
    - 수량 조정 컨트롤: `- [수량 표시] +`.
- 주문 현황 영역
  - 타이틀: `주문 현황`.
  - 주문 리스트는 최신 주문이 상단에 오도록 정렬한다.
  - 각 주문 행 구성
    - 주문 시간 예: `7월 31일 13:00`.
    - 주문 내역 요약 예: `아메리카노 (ICE) × 1`.
    - 주문 금액 예: `4,000원`.
    - 주문 상태 및 상태 변경 버튼 예: `주문 잔료`.

#### 4.3.4 기능 요구사항
- 관리자 대시보드 요약
  - 데이터 항목 예시: `totalOrders`, `receivedOrders`, `inProgressOrders`, `completedOrders`.
  - 주문 상태가 변경될 때마다 해당 수치를 재계산하고 화면에 즉시 반영한다.
- 재고 현황
  - 각 메뉴별 재고 수량을 숫자로 표시한다.
  - `-` 버튼 클릭 시 해당 메뉴 재고 수량을 1 감소(최소 0)시킨다.
  - `+` 버튼 클릭 시 해당 메뉴 재고 수량을 1 증가시킨다.
  - 수량 표시 필드는 초기에는 read-only로 구현하고, 향후 직접 입력 기능으로 확장 가능하다.
  - 재고 변경 시 서버에 업데이트 요청을 보내고, 실패 시 직전 값으로 롤백 및 에러 메시지를 표시한다.
- 주문 현황
  - 주문 리스트 데이터 필드 예시: `orderId`, `orderedAt`, `itemsSummary`, `totalPrice`, `status`.
  - 기본 상태 타입 예: `RECEIVED`, `IN_PROGRESS`, `COMPLETED`, `CANCELED`.
  - 초기 구현에서는 와이어프레임에 맞춰 `주문 잔료` 버튼 한 개로 제공하고, 클릭 시 상태를 `COMPLETED`로 변경한다.
  - 상태 변경 성공 시
    - 해당 주문 행의 상태 및 버튼 UI를 업데이트한다.
    - 상단 관리자 대시보드 요약 수치를 재계산한다.
  - 실패 시
    - 상태를 원래대로 롤백하고 에러 메시지를 표시한다.

#### 4.3.5 UX / UI 요구사항
- `관리자 대시보드`, `재고 현황`, `주문 현황` 세 영역을 카드/박스로 시각적으로 분리한다.
- 재고 수량, 금액 등 숫자는 우측 정렬하며, 중요한 수치는 굵게 표현한다.
- 재고 변경 또는 주문 상태 변경 시, 작은 애니메이션이나 하이라이트로 변경 사실을 사용자에게 알려준다.
- PC에서는 재고 카드 3열, 모바일에서는 1~2열 레이아웃으로 재배치한다.

#### 4.3.6 상태 및 에러 처리
- 재고 데이터 로딩 실패
  - `재고 정보를 불러오지 못했습니다. 다시 시도해 주세요.` 메시지와 재시도 버튼을 제공한다.
- 주문 리스트 로딩 실패
  - `주문 정보를 불러오지 못했습니다. 다시 시도해 주세요.` 메시지를 표시한다.
- 주문 없음 상태
  - `현재 들어온 주문이 없습니다.` 문구를 표시한다.
- 동시성 이슈
  - 여러 단말에서 동시에 재고/주문을 변경할 수 있으므로, 서버 응답 기준으로 최종 상태를 재반영한다.

#### 4.3.7 비기능 요구사항
- 관리자 화면 진입 시 재고 및 주문 리스트 데이터는 1초 이내 로딩을 목표로 한다.
- 재고 조정 및 주문 상태 변경은 낙관적 업데이트(optimistic UI)를 적용하여 체감 속도를 높인다.
- 관리자 화면은 인증된 사용자만 접근할 수 있도록 하고, 상세 권한 모델은 별도 문서에서 정의한다.

## 5. 백엔드 PPD (PostgreSQL / API 설계)

### 5.1 데이터 모델

- **Menus**
  - `id` (PK, bigint, auto increment)
  - `name` (varchar) : 커피 이름 (예: 아메리카노 ICE)
  - `description` (text) : 커피 설명
  - `price` (integer) : 기본 가격 (원 단위)
  - `image_url` (varchar) : 이미지 URL
  - `stock_quantity` (integer) : 재고 수량
  - `created_at` (timestamp with time zone, default now())
  - `updated_at` (timestamp with time zone, default now())

- **Options**
  - `id` (PK, bigint, auto increment)
  - `menu_id` (FK → Menus.id) : 어떤 메뉴에 연결된 옵션인지
  - `name` (varchar) : 옵션 이름 (예: 샷 추가)
  - `extra_price` (integer) : 옵션 추가 금액
  - `created_at` (timestamp with time zone, default now())
  - `updated_at` (timestamp with time zone, default now())

- **Orders**
  - `id` (PK, bigint, auto increment)
  - `ordered_at` (timestamp with time zone, default now()) : 주문 일시
  - `status` (varchar) : 주문 상태 (`RECEIVED`, `IN_PROGRESS`, `COMPLETED`)
  - `total_price` (integer) : 주문 전체 금액 합
  - `created_at` (timestamp with time zone, default now())
  - `updated_at` (timestamp with time zone, default now())

- **OrderItems**
  - `id` (PK, bigint, auto increment)
  - `order_id` (FK → Orders.id)
  - `menu_id` (FK → Menus.id)
  - `quantity` (integer) : 수량
  - `unit_price` (integer) : 메뉴 1개 기준 가격
  - `line_price` (integer) : (메뉴 + 옵션) × 수량 금액

- **OrderItemOptions**
  - `id` (PK, bigint, auto increment)
  - `order_item_id` (FK → OrderItems.id)
  - `option_id` (FK → Options.id)
  - `option_name` (varchar) : 주문 시점 옵션 이름 스냅샷
  - `extra_price` (integer) : 옵션 추가 금액 스냅샷

### 5.2 사용자 흐름 관점 데이터 플로우

1. **메뉴 조회**
   - 프런트의 `주문하기` 화면 진입 시 백엔드에서 `Menus` 리스트를 조회한다.
   - 메뉴 정보 중 `stock_quantity` 는 **관리자 화면에서만 표시**하고, 주문 화면에는 노출하지 않아도 된다(추후 품절 표시 등은 옵션).

2. **장바구니 담기**
   - 사용자가 커피 메뉴 + 옵션을 선택하면 프런트의 장바구니 상태에만 반영한다.
   - 이 단계에서는 DB에 아무 것도 쓰지 않고, `Menus`, `Options` 데이터는 참고용으로만 사용한다.

3. **주문 생성**
   - 사용자가 장바구니에서 `주문하기` 버튼을 누르면 다음 정보를 바탕으로 Orders 관련 테이블에 저장한다.
     - `Orders`
       - `ordered_at` : 현재 시각
       - `status` : 기본값 `RECEIVED` (주문 접수)
       - `total_price` : 장바구니 전체 합계
     - `OrderItems`
       - 각 장바구니 라인에 대해 한 행 생성 (메뉴, 수량, 단가, 라인 합계)
     - `OrderItemOptions`
       - 각 장바구니 라인의 옵션들에 대해 한 행씩 생성 (옵션 이름, 추가 금액)

4. **관리자 화면에 주문 표시 / 상태 변경**
   - 관리자 화면의 `주문 현황`은 `Orders` + `OrderItems` 요약 정보를 조회해서 표시한다.
   - 주문의 기본 상태는 `RECEIVED` (주문 접수)이며,
     - 관리자가 `제조 시작` 버튼을 누르면 `status` 를 `IN_PROGRESS` 로 변경.
     - 이후 `제조 완료`(또는 완료 버튼)를 누르면 `status` 를 `COMPLETED` 로 변경.
   - 상태 변경 시 `updated_at` 을 함께 갱신한다.

5. **재고 반영**
   - 주문이 **성공적으로 생성**된 후, 각 `OrderItems` 의 `quantity` 만큼 `Menus.stock_quantity` 를 차감한다.
   - 재고가 0 아래로 내려가지 않도록 서버에서 검증한다.

### 5.3 API 설계

#### 5.3.1 메뉴 조회 API

- **GET `/api/menus`**
  - **설명**: `주문하기` 화면에서 커피 메뉴 목록을 불러올 때 사용.
  - **쿼리 파라미터(옵션)**:
    - `includeOptions` (boolean, default: true) : true 일 때 각 메뉴의 옵션 목록까지 함께 반환.
  - **응답 예시**
    - 200 OK
    - `[{ id, name, description, price, imageUrl, stockQuantity, options: [{ id, name, extraPrice }] }]`

#### 5.3.2 주문 생성 API

- **POST `/api/orders`**
  - **설명**: 사용자가 장바구니에서 `주문하기`를 눌렀을 때 주문을 생성.
  - **요청 바디 예시**
    - `menus`/`options` 의 id 를 기준으로 요청:
      ```json
      {
        "items": [
          {
            "menuId": 1,
            "quantity": 2,
            "optionIds": [1, 2]
          }
        ]
      }
      ```
  - **서버 처리**
    - 각 `menuId` 에 대해 현재 `stock_quantity` 를 확인한다.
    - 재고가 부족한 경우 에러(409 Conflict 등) 반환.
    - 충분한 경우:
      1. `Orders` 레코드 생성 (`status = RECEIVED`).
      2. 각 아이템별 `OrderItems` + `OrderItemOptions` 생성.
      3. `Menus.stock_quantity` 감소.
  - **응답 예시**
    - 201 Created
    - `{ "orderId": 123, "status": "RECEIVED" }`

#### 5.3.3 주문 상세 조회 API

- **GET `/api/orders/:orderId`**
  - **설명**: 특정 주문 ID 에 대한 전체 정보를 조회.
  - **응답 예시**
    - 200 OK
    - ```json
      {
        "id": 123,
        "orderedAt": "2026-03-17T10:00:00Z",
        "status": "RECEIVED",
        "totalPrice": 12000,
        "items": [
          {
            "menuName": "아메리카노 (ICE)",
            "quantity": 2,
            "unitPrice": 4000,
            "linePrice": 8000,
            "options": ["샷 추가", "시럽 추가"]
          }
        ]
      }
      ```

#### 5.3.4 주문 목록 / 상태 변경 API

- **GET `/api/orders`**
  - **설명**: 관리자 화면의 `주문 현황` 리스트를 위한 API.
  - **쿼리 파라미터(옵션)**:
    - `status` (예: RECEIVED, IN_PROGRESS 등)
    - `limit`, `offset` : 페이지네이션

- **PATCH `/api/orders/:orderId/status`**
  - **설명**: 주문 상태를 `RECEIVED → IN_PROGRESS → COMPLETED` 로 변경.
  - **요청 바디 예시**
    - `{ "status": "IN_PROGRESS" }`
  - **응답 예시**
    - 200 OK, 변경된 주문 정보 반환.

#### 5.3.5 재고 관리 API

- **PATCH `/api/menus/:menuId/stock`**
  - **설명**: 관리자 화면에서 `- / +` 를 눌렀을 때 재고 수량을 조정.
  - **요청 바디 예시**
    - `{ "delta": -1 }` 또는 `{ "stockQuantity": 10 }`
  - **응답**
    - 200 OK, 변경된 `stockQuantity` 반환.
